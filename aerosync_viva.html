Problem & Vision Q1 – Q3
Existing airline scheduling systems like Sabre and Amadeus are monolithic, batch-oriented, and siloed — flight scheduling and cargo management live in separate systems with no real-time sync. AeroSync addresses this by unifying both into a single WebSocket-driven platform with live disruption awareness.

The key gap we fill is real-time collaborative visibility: any gate change or cargo overload is broadcast to all ops clients within 200ms, eliminating the phone-call-and-spreadsheet workflows still common in regional airlines.
mention the 200ms latency target and the "single pane of glass" concept — examiners love concrete metrics
Three primary personas:

1. Flight Dispatchers — need sub-second awareness of gate changes and delays across 20+ concurrent flights.
2. Cargo Managers — need real-time capacity utilization to prevent overloading and revenue loss from empty cargo slots.
3. Operations Leads — need to simulate disruption scenarios before they happen to pre-position resources.

All three currently rely on disconnected tools: separate OMS, CMS, and weather dashboards with no unified view.
frame each persona around a "job to be done" — what they're trying to accomplish, not just who they are
FlightAware is a tracking tool, not a scheduling tool — it shows you where aircraft are, not how to optimise their schedules. Amadeus is enterprise-grade but costs millions in licensing, requires months of integration, and has no AI suggestion layer.

AeroSync differentiates on three axes:
1. Real-time WebSocket architecture vs batch polling
2. Integrated AI recommendations with explainability ("Why this suggestion?")
3. Disruption simulation — no competitor lets you inject a hypothetical event and immediately see cascade impact across your entire network.
the disruption simulator is your strongest differentiator — no existing tool has this feature
Tech Stack Q4 – Q8
Create React App is deprecated and uses Webpack which is significantly slower. Next.js adds server-side rendering complexity we don't need — AeroSync is a real-time ops dashboard, not a content site, so SSR provides no SEO benefit.

Vite uses ESBuild for bundling, giving sub-300ms hot module replacement during development. React 18 specifically gives us concurrent rendering features — useTransition and Suspense — which let us update the Gantt board without blocking WebSocket renders, keeping the UI responsive under high event load.
key phrase: "concurrent rendering" — shows you understand React 18's architecture, not just its API
Raw WebSockets require manual implementation of reconnection logic, room management, and fallback transports. Socket.IO v4 provides automatic reconnection with exponential backoff, named event namespaces (so flight:updated and cargo:updated don't collide), and graceful HTTP long-polling fallback for restrictive corporate networks.

Server-Sent Events are unidirectional — the client can't push disruption injection events back to the server. Socket.IO's room model also lets us broadcast only to clients viewing a specific airport hub, reducing unnecessary traffic.
SSEs being unidirectional is a killer fact — it shows you understand the protocol, not just the library
Flask is synchronous by default — each prediction request blocks the thread. FastAPI is built on Starlette and uses Python's async/await natively, meaning it can handle multiple concurrent prediction requests without threading overhead.

FastAPI also auto-generates OpenAPI docs at /docs, invaluable for testing the delay prediction endpoint. Django is a full-stack framework — far too heavy for a microservice that just serves ML model inference. FastAPI with Uvicorn benchmarks at roughly 3-4x the throughput of equivalent Flask apps.
mention ASGI vs WSGI — FastAPI uses ASGI (async), Flask uses WSGI (sync) — this is the core architectural difference
Leaflet and Google Maps are 2D map libraries — they can't render a rotatable 3D globe with flight arcs as animated 3D polylines. CesiumJS is purpose-built for geospatial 3D visualization using WebGL, which means flight paths are rendered as glowing arcs with accurate great-circle routing (the actual shortest path between two airports), not straight lines on a flat projection.

It also supports time-dynamic data natively through its CZML format, which aligns perfectly with our WebSocket flight position updates.
great-circle routing is a technically impressive detail — it shows you know the geography behind aviation
We use a Random Forest classifier from scikit-learn. Random Forests are an ensemble method — they train multiple decision trees on random subsets of features and aggregate predictions, which reduces overfitting compared to a single decision tree.

Input features: scheduled departure time, origin/destination airport pair, day of week, historical delay rate for that route, and a weather severity score. Random Forest handles the mix of categorical (airport codes) and numerical (weather score) features well without extensive preprocessing. It also provides feature importances, which power the "Why this suggestion?" explainability panel in the UI.
always mention explainability (XAI) — Random Forest feature importance is why you chose it over a black-box neural network
Architecture & Code Q9 – Q12
AeroSync is a three-tier microservice architecture. The React frontend communicates with the Node.js backend via two channels: REST (HTTP) for CRUD operations like fetching flight lists, and WebSocket for real-time push events.

The Node.js backend communicates with the Python FastAPI AI service via internal HTTP — when a dispatcher requests a schedule suggestion, Node proxies the request to FastAPI's POST /predict/delay endpoint. The backend runs a setInterval every 3-5 seconds that mutates mock flight state and emits flight:updated events to all Socket.IO clients, simulating a live data feed.
draw this on the whiteboard if asked — Client → Node (REST + WS) → FastAPI (HTTP) is the communication triangle
1. Server: flightSocket.js holds a setInterval that picks a random flight, mutates its status (e.g. on-time → delayed), and calls io.emit('flight:updated', payload).

2. Client hook: The useWebSocket custom hook registers a listener: socket.on('flight:updated', handler).

3. State update: The handler dispatches to global state (Context/Zustand), updating the matching flight object.

4. Re-render: All subscribed components — Globe arcs, Gantt blocks, HUD stats — re-render simultaneously in the same React render cycle, giving true multi-panel synchronization.
emphasise "same render cycle" — this is why WebSockets beat polling: all panels update atomically, not one by one
The repo uses a monorepo with three top-level directories: /frontend, /backend, and /ai-service. Monorepo was chosen because a single PR can include a frontend feature, the backend endpoint it calls, and the AI model change it depends on — keeping changes atomic and reviewable together.

Each service has its own package.json or requirements.txt so they can be deployed independently. The /docs folder contains the PRD and API contracts, which serve as the source of truth for all inter-service communication.
contrast with polyrepo — monorepo trades deployment independence for easier cross-service refactoring
The user selects a disruption type and airport, then clicks "Inject Disruption". The frontend emits a disruption:inject WebSocket event. The server handler queries all flights departing from/arriving at the affected airport within the next 4 hours, applies a delay multiplier based on severity (1–10 scale), and emits a disruption:cascade event back to all clients.

The payload contains an array of affected flight IDs with estimated delay deltas. The React cascade visualization renders this as a D3-powered radial diagram where affected flights radiate outward from the disruption point with line thickness proportional to delay magnitude.
the D3 radial diagram is a strong technical detail — mention that line thickness is data-encoded (not decorative)
Frontend & UX Q13 – Q16
The Gantt board uses the HTML5 Drag and Drop API. Each flight block is a div with draggable="true" and onDragStart handler that stores the flight ID and current time slot in the drag event's data transfer object.

The time slot cells are drop targets with onDragOver (which calls preventDefault() to allow dropping) and onDrop handlers. On drop, a PATCH /api/flights/:id REST call updates the schedule, and a schedule:update WebSocket event is emitted so all other connected dispatchers see the change immediately. Framer Motion handles the visual snap animation.
remember: onDragOver must call preventDefault() — without it the drop event never fires. This is a classic gotcha
Glassmorphism is a design aesthetic using semi-transparent panels with backdrop-filter: blur() to create a frosted glass effect, layered over darker or more complex backgrounds.

We chose it because aviation operations interfaces are data-dense — glassmorphism panels allow the 3D globe to remain visible beneath overlaid data panels, maintaining spatial context while reading flight data. CSS implementation: background: rgba(10,15,30,0.6); backdrop-filter: blur(12px); border: 1px solid rgba(255,255,255,0.08). The dark navy (#0A0F1E) with cyan (#00E5FF) accents is deliberately inspired by radar and mission control interfaces.
link the design choice to a functional reason — "maintains spatial context" is much stronger than "it looks cool"
Global flight state lives in a React Context (or Zustand store) at the app root level. The useWebSocket hook connects to Socket.IO and dispatches incoming events as state actions — a flight:updated event triggers a state update that replaces the matching flight object in the flights array.

All four screens subscribe to this store via custom hooks like useFlightData(). Local UI state (filters, drag state, selected items) stays in component-level useState — only shared cross-screen data lives in global state, keeping the architecture clean and avoiding unnecessary re-renders.
the key insight: global state = shared data, local state = UI concerns. This separation is what makes the app performant
When the Scheduling Board mounts, it sends POST /api/predict/delay for each flight in view. FastAPI runs the Random Forest model and returns a delay probability and suggested action.

Each suggestion card shows: the recommendation text, a confidence score bar (CSS width proportional to the 0–1 probability), and a "Why?" toggle that expands to show the top 3 feature importances from the model (e.g. "Weather score: 0.72 at LHR contributed 41% to this prediction").

This is Explainable AI (XAI) — critical for any system where humans must trust and act on machine recommendations.
XAI is a hot topic — say "we used Random Forest specifically because it gives us interpretable feature importances, unlike a neural network"
Deployment & Scalability Q17 – Q20
Three specific issues we encountered and resolved:

1. package.json not found — Render was looking in the repo root, but the frontend is in /frontend. Fixed by setting Root Directory to frontend in Render settings.

2. vite: not found (exit 127) — Render skips devDependencies in production installs. Fixed by moving Vite from devDependencies to dependencies.

3. Blocked request error — Vite's preview server has an allowedHosts security check. Fixed by adding preview.allowedHosts: ['aerosync-td50.onrender.com'] to vite.config.js.
these are real bugs you actually debugged — own them confidently, they show practical DevOps experience
Four scaling steps:

1. Database: Replace mock data with PostgreSQL + indexed queries on flight status and departure time.
2. Redis adapter: Add Redis as a Socket.IO adapter — allows multiple Node.js instances to share WebSocket state, enabling horizontal scaling behind a load balancer.
3. AI scaling: Move FastAPI to a dedicated GPU instance or use a managed ML endpoint (AWS SageMaker / Google Vertex AI).
4. CDN: Cloudflare in front of the static React bundle.

WebSocket connections are the hardest to scale — each persistent connection consumes server resources, so Redis pub/sub is the critical architectural addition.
Redis as Socket.IO adapter is the power answer here — it's the specific solution for scaling stateful WebSocket servers
Currently implemented: Helmet.js (secure HTTP headers: X-Frame-Options, CSP, X-XSS-Protection), CORS restricted to frontend domain, .env files excluded from git.

For production I would add:
1. JWT authentication with refresh token rotation for all API and WebSocket connections
2. Rate limiting on the disruption injection endpoint to prevent abuse
3. Input validation using Zod or Joi on all REST payloads
4. HTTPS — Render provides this via Let's Encrypt automatically
5. WebSocket message auth — currently any client can emit disruption:inject, which should require a valid token
point 5 (unauthenticated WS events) shows security-minded thinking — examiners are impressed when you identify your own vulnerabilities
Three honest improvements:

1. Database-first: Start with PostgreSQL + Prisma ORM instead of mock data — retrofitting a DB schema onto an existing codebase is painful and forces architectural compromises.

2. TypeScript from day one: The WebSocket event payloads and REST response types are the exact kind of data contracts that benefit from compile-time type checking. A flight:updated shape mismatch between server and client is very hard to debug at runtime.

3. Separate REST and WebSocket servers: They have different scaling characteristics and failure modes — running them as separate services (different ports or containers) makes the system more resilient and independently deployable.
this is the most important question — a mature, honest answer with specific technical reasoning impresses more than "nothing, it's perfect"
